1. Vue d'ensemble de l'architecture

L'application est structurée en couches indépendantes, chacune avec une seule responsabilité. Ce principe s'appelle la séparation des responsabilités (Separation of Concerns) — c'est un standard de l'industrie logicielle.

datasets/ ← Fichiers Excel source (données brutes)
    └──→ data_loader.py ← Lecture & mise en cache
         └──→ feature_engineering.py ← Calcul des 22 indicateurs
              └──→ ml_model.py ← XGBoost + scores de risque
                   └──→ charts.py ← 15 graphiques Plotly
                        └──→ pages/ ← 5 onglets de l'interface
                              └──→ app.py ← Orchestrateur principal
"L'application est structurée en couches. Les données arrivent via data_loader, sont transformées par feature_engineering, le modèle tourne dans ml_model, les graphiques sont dans charts, et chaque onglet a sa propre page. Si une couche change — par exemple on remplace XGBoost par un réseau de neurones — seul le module ml_model est touché. C'est ce qu'on appelle une architecture modulaire."
Avantage concret : Un changement dans une couche n'impacte pas les autres. L'application est maintenable, testable, et évolutive.

2. app.py — Le chef d'orchestre

Ce qu'il fait

C'est le point d'entrée de l'application. Quand vous lancez streamlit run app.py, c'est ce fichier qui démarre tout.

Contenu

  • La configuration de la page (titre, icône, layout large)
  • Le chargement des données depuis les fichiers Excel
  • La barre latérale avec les filtres (type d'équipement, niveau de coût, période)
  • L'appel au pipeline ML (entraînement du modèle)
  • La création des 5 onglets et l'appel à chaque page

Pipeline principal

def run_pipeline(df_equip, df_maint):
    eq_stats      = compute_equipment_stats(df_maint, df_equip)
    X, y          = prepare_ml_features(eq_stats)
    model_res     = build_model(X, y)
    eq_stats_full = enrich_equipment_stats(eq_stats, model_res["proba"])
    return eq_stats_full, model_res

Pourquoi cette conception

Ce fichier est intentionnellement vide de toute logique métier. Il ne fait qu'orchestrer — appeler les autres modules dans le bon ordre. C'est le principe de séparation des responsabilités : si demain vous changez le modèle ML, vous touchez uniquement ml_model.py, pas app.py.

"L'application démarre ici. Ce fichier est le chef d'orchestre : il charge les données, applique les filtres de l'utilisateur, lance l'entraînement du modèle, puis distribue l'affichage aux 5 onglets. Lui-même ne calcule rien — il délègue tout."

"Tout l'accès aux fichiers Excel est isolé ici. Si demain les données viennent d'une base SQL ou d'une API, on ne modifie que ce fichier — le reste de l'application ne change pas. Le système de cache évite de relire les fichiers à chaque interaction utilisateur."

5. modules/feature_engineering.py — La transformation des données

Ce qu'il fait

Transforme l'historique brut de maintenance en 22 indicateurs métier exploitables par le modèle ML. C'est le module le plus important de l'application. Les variables se répartissent en deux groupes : 12 variables issues directement du fichier équipements, et 10 variables calculées à partir de l'historique de maintenance.

Groupe 1 — Variables issues du fichier équipements (12 variables)

Ces variables décrivent les caractéristiques intrinsèques et l'environnement opérationnel de chaque équipement. Elles sont présentes directement dans le fichier Excel sans calcul supplémentaire.

Pourquoi deux méthodes d'encodage différentes ?
Les variables Low/Normal/High ont un ordre naturel (High > Normal > Low) — on utilise l'encodage ordinal (0/1/2) pour que le modèle respecte cet ordre.
Le type d'équipement (Compresseur, Pompe, Moteur…) n'a pas d'ordre naturel — on utilise LabelEncoder qui attribue un entier arbitraire sans imposer de relation d'ordre.

Composante du score de risque (poids 15%).

Retourne 9 999 si aucune panne connue.		 Combiné au MTBF, indique si l'équipement est "en sursis" (days_since_last_cm > MTBF) ou s'il vient d'être réparé. Permet de calculer le facteur de dépassement MTBF dans le score de risque.

Calcul du MTBF — détail technique

Si ≥ 2 pannes :
    cm_triées = historique_CM trié par date croissante
    intervalles = différences entre dates consécutives (en secondes / 86400)
    MTBF = moyenne(intervalles)

Si 1 panne exactement :
    MTBF = durée_totale_observation (en années × 365,25)
    → Logique : si l'équipement n'a eu qu'une panne en 3 ans, son MTBF ≥ 3 ans

Si 0 panne :
    MTBF = 9 999 jours (valeur sentinelle = "très fiable")
    → Ce MTBF n'est pas utilisé pour prédire une panne — il signale l'absence de risque connu.

Variable cible du modèle ML

La variable cible (ce que le modèle doit prédire) est construite à partir de cm_per_year :

médiane = médiane de cm_per_year sur l'ensemble du parc
target = 1  si cm_per_year > médiane   →  "haute fréquence de pannes"
target = 0  sinon                      →  "fréquence normale ou faible"

Ce seuil médian est choisi pour obtenir des classes équilibrées (50/50 environ) — condition favorable à l'apprentissage du modèle.

Pourquoi 22 variables et pas plus ? Avec seulement 80 équipements, multiplier les variables augmente le risque de surapprentissage (le modèle mémorise au lieu d'apprendre). 22 variables bien sélectionnées donnent de meilleurs résultats qu'une centaine de variables redondantes.
"Ce module fait le travail le plus important : transformer 3 ans d'historique de pannes en 22 indicateurs numériques que le modèle peut comprendre. Par exemple, il ne donne pas juste 'cet équipement a eu 15 pannes' — il calcule '4,8 pannes/an, MTBF de 63 jours, 3 pannes dans les 6 derniers mois, 70% de pannes critiques' — une information beaucoup plus riche qui capte à la fois la fréquence, la sévérité, la tendance récente et l'environnement opérationnel."

6. Variable cible, métriques & score de risque — Questions clés

6.1 La variable cible — Qu'est-ce que le modèle cherche à prédire ?

Le modèle XGBoost résout un problème de classification binaire supervisée. Il apprend à distinguer deux catégories d'équipements :

Valeur cibleSignificationCritère
1 Équipement à haute fréquence de pannes cm_per_year > médiane du parc
0 Équipement à fréquence normale ou faible cm_per_year ≤ médiane du parc 
médiane = médiane de cm_per_year sur les 80 équipements
target = 1  si cm_per_year > médiane   →  "haute fréquence de pannes"
target = 0  sinon                       →  "fréquence normale ou faible"

Pourquoi avoir choisi cette variable cible ?

  • Disponibilité : cm_per_year est calculable directement depuis l'historique — pas besoin d'étiquetage manuel.
  • Pertinence métier : La fréquence de pannes est l'indicateur le plus direct du coût de maintenance et du risque opérationnel.
  • Classes équilibrées : En utilisant la médiane comme seuil, on obtient exactement 50% de "1" et 50% de "0" — condition idéale pour l'apprentissage, qui évite le biais vers la classe majoritaire.

Pourquoi la médiane et non un seuil fixe ?

Un seuil fixe (ex: "plus de 5 pannes/an") dépendrait du parc spécifique. Si le parc change (nouveaux équipements plus fiables), le seuil deviendrait obsolète. La médiane s'adapte automatiquement à la composition du parc — elle classe toujours les 50% d'équipements les plus défaillants comme "à risque élevé".

Ce que le modèle NE prédit PAS : il ne prédit pas une date exacte de panne. Il identifie les équipements structurellement plus susceptibles de tomber en panne fréquemment. La date de prochaine panne est calculée séparément via le MTBF.

6.2 Les métriques d'évaluation — Pourquoi ces indicateurs ?

Deux métriques principales ont été choisies pour évaluer les performances du modèle :

MétriqueCe qu'elle mesureRésultat obtenuPourquoi ce choix
Précision (Accuracy) Proportion d'équipements correctement classés (classe 0 et classe 1 confondues) ~95% Indicateur global simple, compréhensible par tous. Valable ici car les classes sont équilibrées (50/50).
AUC-ROC Capacité du modèle à distinguer les vrais positifs des faux positifs, pour tous les seuils de décision possibles. Une valeur de 1.0 = discrimination parfaite, 0.5 = modèle aléatoire. ~0.990 (test) ~0.985 (CV) Métrique standard en classification médicale et industrielle. Plus robuste que la précision car indépendante du seuil de décision choisi.

Pourquoi l'AUC-ROC plutôt que la précision seule ?

La précision peut être trompeuse : un modèle qui prédit toujours "0" sur un dataset déséquilibré obtiendrait 80% de précision sans rien apprendre. L'AUC-ROC mesure la qualité intrinsèque de la séparation entre classes. Ici, AUC = 0.990 signifie que si on choisit aléatoirement un équipement à haute fréquence et un équipement à basse fréquence, le modèle identifie correctement lequel est lequel dans 99% des cas.

Pourquoi la cross-validation 5 folds ?

Avec seulement 80 équipements, un simple split 75/25 risque de dépendre du hasard de la répartition. La cross-validation divise le dataset en 5 groupes égaux (16 équipements chacun), entraîne le modèle sur 4 groupes et teste sur le 5ème — puis répète 5 fois en changeant le groupe de test. Résultat : chaque équipement est testé exactement une fois, et on obtient 5 scores indépendants dont on calcule la moyenne et l'écart-type.

Indicateur CVValeurInterprétation
AUC-ROC moyen (5 folds)~0.985Performance stable sur l'ensemble du parc
Écart-type (±)~0.010Faible variance = pas de surapprentissage sur un sous-ensemble
Règle simple : Si AUC(test) ≈ AUC(CV) avec un faible écart-type → le modèle généralise bien. Si AUC(test) ≫ AUC(CV) → il a mémorisé les données d'entraînement (surapprentissage). Notre cas : AUC test ≈ AUC CV → modèle sain.

6.3 Probabilité ML vs Score de risque — Quelle différence ?

Ce sont deux choses fondamentalement différentes, même si elles sont liées.

Probabilité MLScore de risque composite
Nature Sortie statistique du modèle XGBoost Indicateur métier multi-dimensionnel
Plage de valeurs 0 à 1 (probabilité) 0 à 100 (score lisible)
Ce qu'il mesure La probabilité que l'équipement appartienne à la classe "haute fréquence de pannes" — selon les 22 patterns appris Le risque global de l'équipement en combinant plusieurs dimensions métier
Basé sur Les 22 variables d'entrée, traitées par les 150 arbres XGBoost 7 composantes dont la probabilité ML (30%), mais aussi le downtime, les pannes récentes, la criticité…
Avantage Signal objectif, appris depuis les données Intègre des dimensions que le modèle seul ne capture pas (impact opérationnel, tendance récente)
Limite "Boîte noire" — difficile à expliquer intuitivement Dépend des pondérations choisies — subjectivité assumée
Utilisé pour Alimentation du score composite (30%) + graphique "Probabilité par équipement" Classement des équipements, alertes, tableau de planification, recommandations
"La probabilité ML dit : 'statistiquement, cet équipement ressemble aux équipements qui tombent souvent en panne'. Le score de risque dit : 'en tenant compte de tout — sa fréquence, son downtime, ses pannes récentes, sa PM — quel est son niveau de danger réel pour l'exploitation ?' Ce sont deux angles complémentaires."

6.4 Comment le score de risque est-il calculé ?

Le score est une somme pondérée de 7 composantes normalisées, multiplié par 100 pour obtenir un score lisible de 0 à 100.

Étape 1 — Normalisation min-max de chaque composante

Chaque composante est ramenée sur une échelle commune [0, 1] via la formule :

valeur_normalisée = (valeur − min_du_parc) / (max_du_parc − min_du_parc)

Cela permet de comparer des grandeurs d'unités différentes (heures de downtime, nombre de pannes, ratio…) sur la même échelle.

Étape 2 — Calcul du facteur de dépassement MTBF

Ce facteur mesure si l'équipement est "en sursis" — c'est-à-dire si le temps écoulé depuis la dernière panne dépasse déjà son MTBF habituel :

overdue = (jours_depuis_dernière_panne / MTBF).clip(0, 2) / 2

On le plafonne à 2× le MTBF (dépassement au-delà = score maximal sur cette composante).

Étape 3 — Combinaison pondérée

score_brut =
    0.30 × probabilité_ML
  + 0.20 × normalize(cm_per_year)          ← fréquence annuelle de pannes
  + 0.15 × normalize(cm_last_6m)           ← tendance des 6 derniers mois
  + 0.10 × overdue                         ← dépassement du MTBF
  + 0.10 × normalize(high_crit_ratio)      ← sévérité des pannes
  + 0.10 × normalize(total_downtime_h)     ← impact opérationnel cumulé
  + 0.05 × normalize(1 − pm_compliance)   ← non-respect de la PM

score_final = (score_brut × 100).clip(0, 100)

Étape 4 — Classification en niveaux

NiveauSeuilAction recommandée
🔴 CritiqueScore ≥ 65Intervention immédiate
🟠 Modéré38 ≤ Score < 65Planifier dans les 30 jours
🟢 FaibleScore < 38Surveillance normale

Justification des pondérations

Pourquoi la somme fait 100% ? C'est une contrainte volontaire pour garder le score interprétable. Si un équipement obtient le maximum sur toutes les composantes, son score sera 100. Chaque poids représente la contribution maximale de cette composante au score final.
"Le score de risque est transparent et ajustable. Si demain votre expert métier estime que le downtime devrait peser plus que la fréquence de pannes, on augmente son poids de 10% à 20% en une ligne de code. C'est l'avantage d'un score explicite sur un modèle boîte noire."

7. modules/ml_model.py — Le moteur prédictif

Ce qu'il fait

Entraîne le modèle XGBoost, calcule les scores de risque composites, et prédit les dates de prochaine panne.

6.1 Le modèle XGBoost

Objectif : Classer chaque équipement comme "haute fréquence de pannes" (au-dessus de la médiane du parc) ou non. C'est une classification binaire supervisée.

ParamètreValeur choisieJustification
n_estimators150Assez d'arbres pour la précision, pas trop pour éviter l'overfitting sur 80 échantillons
max_depth4Arbres peu profonds = moins d'overfitting sur petit dataset
learning_rate0.08Apprentissage lent et stable (0.1 par défaut trop rapide)
subsample0.8Chaque arbre voit 80% des données → régularisation stochastique
colsample_bytree0.880% des features par arbre → diversité des arbres

Validation : Split 75% entraînement / 25% test + cross-validation 5 folds stratifiée

MétriqueRésultatInterprétation
Précision (test)~95%95% des équipements correctement classés
AUC-ROC (test)~0.990Capacité de discrimination quasi-parfaite
AUC-ROC (CV moy.)~0.985Score stable sur 5 folds → pas de surapprentissage

6.2 Le score de risque composite (0–100)

Le score de risque ne se réduit pas à la probabilité ML. Il combine 7 composantes métier :

ComposantePoidsJustification du poids
Probabilité ML (XGBoost)30%Signal prédictif principal, le plus objectif
Fréquence de pannes (cm_per_year)20%Indicateur historique le plus direct
CM récentes — 6 mois (cm_last_6m)15%Capture la tendance de dégradation actuelle
Dépassement du MTBF10%Équipement "en sursis" si déjà au-delà du MTBF moyen
Ratio d'interventions critiques10%Sévérité des défaillances passées
Downtime cumulé10%Impact opérationnel réel mesuré
Non-compliance maintenance préventive5%Facteur aggravant si les PM sont négligées

Seuils de classification :

  • Critique Score ≥ 65 — intervention immédiate requise
  • Modéré Score entre 38 et 64 — planifier dans les 30 jours
  • Faible Score < 38 — surveillance normale

6.3 Prédiction de la prochaine panne

cycles_restants = floor(jours_depuis_dernière_panne / MTBF) + 1
date_prochaine_panne = dernière_panne + (cycles × MTBF)
PM recommandée = prochaine_panne − (30% × MTBF)

La PM est placée à 70% du MTBF — intervention avant la panne estimée avec une marge de sécurité de 30%.

Pourquoi vectoriel et non une boucle ? La version initiale utilisait une boucle while qui calculait les cycles un par un. Si le MTBF était inférieur à 1 jour, la boucle pouvait tourner des millions de fois. Le calcul vectoriel O(1) est instantané, quel que soit le MTBF.
"Le modèle atteint 95% de précision sur notre dataset. Mais on n'utilise pas seulement la prédiction ML — on la combine avec 6 autres indicateurs métier pour un score de risque plus robuste et plus explicable. Un équipement peut avoir une faible probabilité ML mais un downtime énorme — le score composite capture ça. C'est aussi plus facile à expliquer aux équipes terrain."

Correction technique — Gantt chart

Un bug de compatibilité entre Plotly et pandas 2.x causait un crash lors de l'affichage du calendrier Gantt. La ligne de référence temporelle déclenchait un calcul interne qui tentait d'additionner un entier à un objet Timestamp pandas — interdit depuis pandas 2.0.

Solution : Séparation de la ligne (add_vline) et de l'annotation texte (add_annotation), en passant la date comme chaîne ISO au lieu d'un objet Timestamp.

"Tous les graphiques sont interactifs — zoom, survol pour les valeurs exactes, export PNG. Ils sont isolés dans un module dédié : si un graphique a un bug, on corrige une seule fonction sans risquer de casser le reste de l'application."

15. Sécurité & bonnes pratiques

Un audit de sécurité a été conduit sur l'ensemble de l'application. Voici les points vérifiés et les corrections apportées :

CatégorieRisque identifiéCorrection apportée
XSS (Cross-Site Scripting) Les champs equipment_id et equipment_type étaient injectés directement dans du HTML sans échappement Application de html.escape() sur toutes les données avant injection HTML dans reco_card()
Boucle infinie Une boucle while dans la prédiction de panne pouvait tourner indéfiniment si le MTBF était < 1 jour Remplacement par un calcul vectoriel O(1) + plancher max(mtbf, 1.0)
Suppression globale des warnings warnings.filterwarnings("ignore") masquait tous les avertissements, y compris les erreurs critiques Suppression ciblée uniquement pour openpyxl et xgboost (warnings connus et inoffensifs)
Dépendances non fixées Sans versions dans requirements.txt, une mise à jour automatique peut casser l'application Toutes les versions fixées dans requirements.txt
Chemins de fichiers Risque d'accès à des fichiers arbitraires si le chemin était paramétrable Chemins codés en dur dans data_loader.py, aucun input utilisateur pour les chemins
Désérialisation unsafe Utilisation de pickle pour sauvegarder des modèles ML = vecteur d'exécution de code arbitraire Aucun pickle dans l'application — le modèle est réentraîné à chaque session (cache Streamlit)

Résumé — Points clés à retenir pour la présentation

Question probableRéponse courte
Pourquoi XGBoost ?Excellent sur les petits datasets tabulaires, gère les features mixtes (numériques + catégorielles encodées), fournit une importance des variables explicable. AUC = 0.990 confirme le bon choix.
Pourquoi seulement 80 équipements ?C'est le parc réel. La cross-validation 5-folds compense la petite taille en utilisant 100% des données pour l'évaluation.
Comment sont déterminés les seuils de risque ?Calibrés sur la distribution des scores (p75 ≈ Critique, p40 ≈ Modéré) et validés avec l'historique connu. Ils sont ajustables dans settings.py.
L'app est-elle sécurisée ?Oui. Audit réalisé : XSS corrigé, pas de pickle, pas de chemins utilisateur, dépendances fixées.
Peut-on la connecter à une vraie base de données ?Oui. Seul data_loader.py doit être modifié — le reste de l'application est indépendant de la source de données.
Comment mettre à jour les données ?Remplacer les fichiers Excel dans datasets/ et relancer l'application — le cache se recharge automatiquement.
