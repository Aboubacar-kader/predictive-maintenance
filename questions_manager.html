💼 Démarche & valeur métier
Stratégie Pourquoi de la maintenance prédictive plutôt que préventive classique ? ▼
La maintenance préventive intervient selon un calendrier fixe — elle peut intervenir trop tôt (gaspillage de ressources) ou trop tard (panne déjà survenue). La maintenance prédictive utilise les données réelles de chaque équipement pour intervenir au bon moment. Résultat : moins de downtime non planifié et meilleure allocation des ressources de maintenance.
Appuyez sur le graphique "Évolution du ratio préventif mensuel" dans l'onglet Rapports pour illustrer la tendance.
ROI Quel est le gain concret pour l'entreprise ? ▼
On identifie les équipements à risque avant la panne. Le tableau de planification indique exactement quand intervenir. Cela permet de :
  • Réduire le downtime non planifié
  • Prioriser les budgets de maintenance sur les équipements qui en ont vraiment besoin
  • Anticiper les besoins en pièces de rechange et en techniciens
Montrez le KPI "Downtime total" dans l'onglet Vue d'ensemble et les équipements du Top 15 dans l'onglet Rapports.
Données Sur quelles données repose l'analyse ? ▼
Deux sources de données :
  • Inventaire des équipements : caractéristiques physiques, conditions opérationnelles, âge, type
  • Historique de maintenance sur 3 ans : chaque intervention corrective (CM) et préventive (PM) avec dates, durées, types, criticité
Le parc compte 80 équipements, représentant plusieurs milliers d'opérations de maintenance.
🤖 Modèle Machine Learning
Choix algo Pourquoi XGBoost et pas un autre algorithme ? ▼
XGBoost est l'algorithme de référence pour les données tabulaires de taille moyenne. Trois raisons principales :
  • Gère nativement les features mixtes (numériques + catégorielles encodées)
  • Robuste au bruit et aux valeurs manquantes
  • Fournit une importance des variables — ce qui permet d'expliquer les prédictions aux équipes terrain
Sur notre dataset, il atteint AUC = 0.990 — confirmant a posteriori la pertinence du choix.
Taille dataset Avec seulement 80 équipements, le modèle n'est-il pas trop limité ? ▼
C'est une limite réelle et assumée. On la compense par trois mesures :
  • Cross-validation 5 folds : chaque équipement est utilisé à la fois pour l'entraînement et le test — on exploite 100% des données
  • Feature engineering soigné : 22 variables bien construites valent mieux que 5 variables brutes
  • Hyperparamètres conservateurs : max_depth=4 pour limiter le surapprentissage
Si on ajoute de nouveaux équipements au parc, le modèle s'améliorera automatiquement.
Objectif Que prédit exactement le modèle ? ▼
Il classe chaque équipement comme "haute fréquence de pannes" (au-dessus de la médiane du parc) ou non. C'est une classification binaire supervisée — pas une prédiction de date exacte, mais un signal de priorité relative au sein du parc.
La date exacte de prochaine panne est calculée séparément via le MTBF — ce sont deux mécanismes distincts.
Performance AUC de 0.990 — n'est-ce pas suspect ? Est-ce du surapprentissage ? ▼
La question est légitime. On l'a vérifiée de deux façons :
  • Le score CV 5 folds est proche du score test avec un faible écart-type — ce qui confirme la stabilité
  • Les features sont fortement prédictives par nature : elles capturent directement l'historique de fiabilité de chaque machine
Un équipement qui tombe en panne 8 fois/an sera toujours au-dessus de la médiane — la relation est quasi-déterministe, d'où l'AUC élevée.
Explicabilité Quelle est la variable la plus importante du modèle ? ▼
Le graphique d'importance des variables dans l'onglet Modèle prédictif le montre précisément. En règle générale, les variables qui dominent sont :
  • cm_per_year — fréquence annuelle des pannes
  • mtbf_days — temps moyen entre pannes
  • cm_last_6m — tendance récente de dégradation
Ce sont les signaux les plus directs de fiabilité d'un équipement.
Montrez le graphique en direct lors de la présentation — c'est plus percutant que de l'expliquer à l'oral.
📊 Score de risque composite
Conception Pourquoi un score composite si vous avez déjà la probabilité ML ? ▼
La probabilité ML est un signal statistique. Le score composite intègre des dimensions que le modèle seul ne capture pas :
  • Le downtime opérationnel réel (impact business)
  • La criticité des pannes passées (sévérité)
  • Le respect du plan de maintenance préventive (facteur aggravant)
Un équipement peut avoir une probabilité ML modeste mais un downtime énorme — il mérite quand même une priorité élevée.
Paramétrage Comment avez-vous fixé les seuils Critique / Modéré / Faible ? ▼
Ils sont calibrés sur la distribution des scores du parc (percentiles) et validés par rapport à l'historique connu. Ils sont entièrement paramétrables dans le fichier config/settings.py — si les équipes métier veulent les ajuster, c'est une seule ligne à modifier.
Pondération Les pondérations (30%, 20%…) sont-elles justifiées scientifiquement ? ▼
Elles reflètent l'importance relative des signaux selon une logique métier :
  • La probabilité ML porte 30% car c'est le signal le plus objectif et global
  • Les pannes récentes (15%) pèsent plus que la non-compliance PM (5%) car elles indiquent une dégradation en cours
Ces poids peuvent être revus avec les experts métier — c'est l'avantage d'un score transparent vs une "boîte noire".
🏗️ Architecture technique
Outil Pourquoi Streamlit et non Power BI ou Tableau ? ▼
Trois raisons techniques :
  • Streamlit permet d'intégrer directement le modèle ML dans l'interface — Power BI ne peut pas entraîner un XGBoost
  • Tout est en Python, la même langue que le modèle — pas de transfert de données entre outils
  • Déployable sur un serveur interne sans licence tierce (open source)
Sécurité L'application est-elle sécurisée ? ▼
Un audit de sécurité complet a été réalisé. Six catégories vérifiées :
  • Protection XSS sur les données injectées en HTML (html.escape())
  • Suppression d'une boucle infinie potentielle dans le calcul de planning
  • Suppression ciblée des warnings (pas de masquage global)
  • Dépendances fixées dans requirements.txt
  • Chemins de fichiers codés en dur (pas d'input utilisateur)
  • Aucun pickle — le modèle est réentraîné à chaque session via le cache Streamlit
Intégration Peut-on connecter l'application à notre ERP ou GMAO ? ▼
Oui. Le module data_loader.py est le seul point d'accès aux données. Il suffit de remplacer la lecture Excel par une connexion SQL ou API — le reste de l'application est totalement indépendant de la source de données.
C'est précisément l'avantage de l'architecture modulaire choisie.
Maintenance Si on veut changer un seuil ou une couleur, qui touche à quoi ? ▼
Un seul fichier : config/settings.py. Tous les paramètres métier sont centralisés là — seuils de risque, couleurs, pondérations, liste des features. Aucun autre fichier à modifier.
🗺️ Limites & roadmap
Honnêteté Quelles sont les limites actuelles de l'application ? ▼
Trois limites honnêtes :
  • Le modèle est réentraîné à chaque lancement — en production il faudrait un pipeline d'entraînement séparé et planifié
  • 80 équipements est un dataset limité ; les performances s'amélioreront avec plus de données
  • Pas d'authentification utilisateur pour l'instant
Reconnaître les limites démontre la maturité technique — c'est toujours apprécié par un manager.
Évolution Que faudrait-il pour passer en production ? ▼
Cinq étapes :
  • Connexion à la GMAO en temps réel
  • Pipeline d'entraînement automatisé (hebdomadaire ou mensuel)
  • Authentification utilisateur (rôles : viewer, admin)
  • Déploiement sur serveur interne (Streamlit Cloud ou Docker)
  • Monitoring des performances du modèle dans le temps (model drift)
Durabilité Le modèle se dégrade-t-il dans le temps ? ▼
Tout modèle se dégrade si les données évoluent — c'est le phénomène de model drift. Des causes possibles : nouveaux équipements, nouveaux types de pannes, changement de processus. Il faut prévoir :
  • Un réentraînement régulier (mensuel ou trimestriel)
  • Un monitoring de l'AUC en production
  • Des alertes si les performances descendent sous un seuil critique
⚠️ La question piège
Piège Si le modèle dit qu'un équipement est à faible risque et qu'il tombe en panne, que se passe-t-il ? ▼
Le modèle est un outil d'aide à la décision, pas un oracle. Il réduit le risque d'être pris par surprise — il ne l'élimine pas. C'est pourquoi :
  • Le score composite intègre plusieurs signaux indépendants pour plus de robustesse
  • L'expertise des techniciens terrain reste une source d'information complémentaire indispensable
  • Les seuils de risque sont volontairement conservateurs pour limiter les faux négatifs
Aucun modèle prédictif ne garantit 0 panne surprise. L'objectif est de passer d'une maintenance réactive à une maintenance principalement anticipée.
Ne jamais promettre 100% de précision — c'est la réponse qui rassure le plus un manager technique.
